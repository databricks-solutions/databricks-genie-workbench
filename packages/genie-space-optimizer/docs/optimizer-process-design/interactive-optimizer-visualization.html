Genie Space Optimizer

A Scientific Loop For Better Answers.

The optimizer treats a Databricks Genie Space as a system under test. It measures it against a benchmark, diagnoses why specific questions fail, applies bounded interventions, and only keeps changes that demonstrably improve the score.

The Stakes

Most Genie Spaces Are Tuned By Intuition. They Don't Have To Be.

A Genie Space can look impressive in demos and be quietly wrong in production. The classic failure pattern is to edit instructions, glance at a few questions, and ship — until a different question silently breaks weeks later. The optimizer replaces this loop with one that produces evidence, not vibes.

What is — Tune by intuition
Manual prompt edits. No record of what changed or why.
A few "golden questions" tested informally.
Subjective verdict. "Looks better." Ship.
Silent regression discovered weeks later.
→ replace with
What could be — Optimize by evidence
Versioned benchmarks in Unity Catalog. Train + held-out splits.
RCA evidence per failure. Patches start from diagnosis.
Judged re-evaluation. Same scorer panel as baseline.
Accept only proven gains. Roll back the rest.
The optimizer's promise: the Genie Space configuration that ships is the one that was proven against a held-out set of questions, not the one that was last edited.

The Gates

Many Proposals Enter. Few Survive. That's The Point.

The optimizer is skeptical by design. Every patch must pass independent gates before it even reaches re-evaluation, and only patches whose post-arbiter accuracy beats the gain floor are kept.

Many proposals — ideas from RCA evidence ↓ candidates: ~30
① Causal grounding — does an RCA row support this patch?
② Blast radius — does it modify only scoped paths?
③ Patch cap — is the change set under budget?
④ Teaching safety — does any instruction conflict?
⑤ Leakage firewall — is anything from held-out leaking?
⑥ Structural & ⑦ Regression gates — valid + non-breaking?
⇣ surviving patch set
Re-evaluation against the train benchmark
Accepted & snapshotted
Rolled back & recorded
What stops a bad patch? Seven independent gates plus a single acceptance criterion: post-arbiter accuracy ≥ baseline + min_gain_pp. The optimizer rejects most of its own ideas — that's how you trust the few it ships.

Ship It

Prove It Generalizes. Prove It Repeats. Then Promote.

Finalize runs the held-out benchmark and the repeatability passes. Deploy carries the proven configuration — the champion LoggedModel — across to the target Genie Space (same workspace or cross-workspace).

Proof of quality

Held-out evaluation
Run the candidate against questions the lever loop never saw. Catches overfitting before it ships.
Repeatability passes
Run the train benchmark N times (default 3) to measure score stability under retries. Variance is a visible metric.
Review session
An MLflow GenAI dashboard aggregates traces, judgements, and reflection entries for human sign-off.
patch_space_config
Carries the champion config, not the latest edit.

Promotion & deployment

Champion alias
Promoted candidate registered as a LoggedModel; champion alias is the immutable carrier.
deploy_check (read-only)
Validates target dependencies, auth context, and approval state — without touching the target.
deploy_execute
PATCHes the proven configuration to the target Genie Space. Same workspace or cross-workspace; the carrier is identical.
